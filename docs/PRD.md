# 主动智能体 PRD v2

## 1. 文档目标

这份 PRD 只定义产品，不替代后续系统设计。
它回答四件事：

1. 这个产品到底是什么
2. 它解决的核心问题是什么
3. 它的第一性输出是什么
4. 第一阶段怎么验证它值得继续做

---

## 2. 产品一句话

**主动智能体** 是一种围绕“当前工作窗口”运行的 agent：它先对当前窗口产出一个**对用户最合理的下一步假设**，再决定是否提醒、请求接力，或在边界内继续做下去。

它不是“更会执行任务的自动化工具”，也不是“按场景切开的聊天助手”。
它首先是一个**主动判断与主动接力产品**。

---

## 3. 为什么现在值得做

2025–2026 的智能体产品已经把“会执行”推到了新阶段：

OpenAI 已经把 computer use 明确定义为：模型观察界面、返回界面动作，由代码或 harness 执行；Codex 也已经扩展到后台 computer use，可在 Mac 上看到、点击、输入，并支持并行 agent。([OpenAI开发者][1])

Google 一边在 Project Mariner 上推进“浏览器中的多任务 agent”，一边又把“通过录屏教会它任务”的 teach 流程做进了产品帮助文档；Gemini 也直接推出了面向日常事务的 Agent 产品入口。([Google DeepMind][2])

Microsoft 在 Copilot Studio 里已经把 **event trigger** 做成正式能力，允许 agent 在事件发生后自主触发，并继续向多 agent 编排推进。([Microsoft Learn][3])

消费级一侧，Comet、Manus、Genspark 也都把“不是回答，而是代你完成事务”作为产品卖点：Comet 强调 personal assistant 式浏览器，Manus 强调 action engine 和 browser operator，Genspark 强调 Super Agent 和多工具自治工作流。([Perplexity AI][4])

所以，问题已经不再只是“agent 能不能做事”。
新的空位在于：

> **它什么时候该先动，以及先动到什么程度。**

这就是本产品的机会。

---

## 4. 市场上已经存在什么，我们不重复什么

现在的主流产品，大致分成四类：

第一类是 **执行代理**。代表是 Operator、Codex computer use、Project Mariner。它们的核心价值是“给我一个目标，我替你操作软件或网页”。([OpenAI开发者][1])

第二类是 **浏览器/工作空间代理**。代表是 Comet、Manus、Genspark。它们把浏览器、工作区、多工具整合成一个能持续帮你处理任务的环境。([Perplexity AI][4])

第三类是 **企业自治代理**。代表是 Copilot Studio 的事件触发和多 agent 编排。它们强在组织流程、治理和企业事件驱动。([Microsoft Learn][3])

第四类是 **成长型代理**。代表是 Hermes 这类把 self-improving、skill growth、长期记忆放在前台的产品方向。([manus.im][5])

本产品不重复这些产品已经做得很清楚的那一层。
本产品要抓的，是它们共同还没有单独清晰产品化的一层：

# **主动资格层**

也就是：

- 当前窗口里到底有没有一个已经显形的任务
- 现在是不是该由 agent 先动
- 它应该提醒、请求接力，还是直接做
- 它为什么这么判断

---

## 5. 产品核心定义

### 5.1 产品对象

主动智能体的产品对象不是“场景”，也不是“行业任务”。

它的第一对象是：

# **当前窗口**

这里的“当前窗口”指的是用户正在面对、并且最可能承载当前工作意图的那个工作界面。
它可以是浏览器页、编辑器页、文档窗口、表单窗口、后台控制台、IDE 窗口，或者其他能承载明确工作状态的可观察界面。

### 5.2 产品核心动作

主动智能体不是先执行，而是先输出一个对象：

# **当前窗口下一步假设**

这就是产品的第一性输出。

---

## 6. 第一性输出定义

主动智能体面对当前窗口时，首先要给出的不是完整答案，也不是直接执行，而是一个结构化判断对象：

### 当前窗口下一步假设

它至少包含六部分：

1. **当前窗口状态摘要**
   它判断当前窗口正在发生什么，处于什么阶段。

2. **最合理的下一步假设**
   对用户来说，现在最值得接住、最值得推进的下一步是什么。

3. **主动级别**
   这一步应该停留在观察、提醒、请求接力，还是直接接力执行。

4. **置信度**
   它对这个假设的把握有多大。

5. **判断依据**
   为什么是这一步，不是别的一步。

6. **停止条件**
   在什么情况下不该继续，不该升级。

这份输出，才是第一阶段真正该被验证的对象。
不是“内容收尾场景”本身，也不是“浏览器 agent 会不会点按钮”本身。

---

## 7. 五个主动维度

这里保留五个主动维度，但把它们放回产品层，不拿它们替代设计。

### 7.1 预判型主动

系统基于窗口趋势、流程阶段和近期动作，提前准备下一步。
它的价值是减少等待和切换。
现在市面上的部分 agent 已经开始触到这层，例如 Project Mariner 的多任务浏览器执行和 teach-and-repeat 路线，都在往“提前准备后续动作”靠近。([Google DeepMind][2])

### 7.2 守望型主动

系统持续观察当前窗口和邻近信号，一旦任务显形，就把它识别出来。
这是本产品第一阶段的核心维度。

### 7.3 接力型主动

系统判断当前后半段已经足够明确，于是主动把它继续做下去。
这和很多执行型 agent 有交集，但我们的中心不是“能做”，而是“为什么此刻该接”。Operator、Codex computer use、Mariner 都证明了“接力执行”在产品上是可行的。([OpenAI开发者][1])

### 7.4 代办型主动

系统在长期委托和稳定边界内，持续代办某类事务。
Copilot Studio 的事件触发，本质上已经让 agent 在某些业务事件出现后开始自主运行。([Microsoft Learn][3])

### 7.5 变革型主动

系统不只会接力，还会逐渐优化“什么该被接力”“什么时候该升级主动级别”。
Hermes 这类成长型产品在推动这一层。([manus.im][5])

---

## 8. 用户是谁

第一批用户不是“大而全所有人”，而是：

- 重度电脑工作者
- 会在多个窗口之间频繁切换的人
- 经常在“已经做到一半”和“还得手工收尾”之间被拖住的人
- 已经接受 agent 能做事，但对“什么时候该做”还不满意的人

这类用户对下面几种摩擦最敏感：

- 明明活已经出来了，还要再说一次
- 明明已经进入收尾阶段，还要自己手工推进
- 明明系统会做，但它不知道该什么时候接
- 明明是低风险后半段，还得自己当路由器

---

## 9. 用户价值

主动智能体给用户的价值，不是“又多一个自动化工具”，而是四件事：

### 9.1 少一次重新表达

用户不需要为已经显形的任务再次完整开口。

### 9.2 少一段人肉收尾

系统接住明确且低风险的后半段。

### 9.3 少一次任务发现负担

系统先帮用户识别“这里其实已经有活”。

### 9.4 多一层分寸感

系统知道什么时候只提醒，什么时候该请求接力，什么时候可以继续做。

---

## 10. 产品边界

这份 PRD 只锁定产品定义，不替代后续设计。
因此它只定义：

- 当前窗口是核心对象
- 当前窗口下一步假设是核心输出
- 五个主动维度是产品理解框架
- 第一阶段重点是守望型主动 + 接力型主动

它暂时不定义：

- 具体模型结构
- 具体执行器选型
- 具体环境适配方式
- 具体学习机制
- 具体工具协议

这些内容属于后续设计文档。

---

## 11. 第一阶段验证对象

第一阶段不验证“所有主动都成立”。
先验证一件事：

# **agent 对当前窗口给出的“下一步假设”是否稳定、合理、可用。**

也就是说，第一阶段真正的验证对象是输出质量，不是应用场景列表。

### 第一阶段要回答的问题

1. 系统能否对当前窗口形成稳定状态判断
2. 系统能否给出一个对用户来说像样的下一步假设
3. 系统能否为这个假设分配合适的主动级别
4. 系统能否说明为什么是这一步

---

## 12. 第一阶段验证样本

这里故意不用“场景定义产品”，而只用“样本验证输出”。

第一阶段可以从三类窗口样本里取数据：

### 12.1 内容编辑型窗口

例如文档编辑器、文章编辑器、发布前草稿窗口。

### 12.2 表单/流程型窗口

例如后台表单页、配置页、提交页、审批页。

### 12.3 停滞/待续型窗口

例如教程页、操作做到一半的工作页、需要续接的进度页。

这三类只是**输出验证样本桶**，不是产品身份。

---

## 13. 成功标准

### 13.1 输出成功

对当前窗口，系统给出的下一步假设能被用户认为“是此刻最合理的继续方向”。

### 13.2 主动成功

系统给出的主动级别大体符合用户直觉：

- 该提醒时提醒
- 该请求接力时请求
- 该自动继续时继续

### 13.3 产品成功

用户第一次明确感受到：

> **它不是只会等我说，它会在窗口里看见活，并知道现在该怎么接。**

---

## 14. 失败信号

如果出现下面几类情况，说明方向偏了：

- 输出仍然是泛泛建议，不能落到“当前窗口下一步”
- 输出太像聊天总结，而不像行动假设
- 主动级别分配混乱
- 验证对象又退回到“场景是否成立”
- 文档开始替代后续设计，提前写死系统方案

---

## 15. 路线图

### Phase 1：输出定义成立

先把“当前窗口下一步假设”定义清楚，并用样本验证输出质量。

### Phase 2：主动级别成立

让系统不仅会猜下一步，还会区分提醒、请求接力、自动接力。

### Phase 3：单类窗口可用

在一类高频窗口中形成稳定输出和稳定接力。

### Phase 4：多窗口迁移

验证当前窗口输出对象在不同窗口类型中是否还能保持质量。

### Phase 5：成长与长期委托

在输出和级别都稳定后，再向长期代办和成长型主动延伸。

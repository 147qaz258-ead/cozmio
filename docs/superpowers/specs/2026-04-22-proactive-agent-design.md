# 主动智能体 设计文档

## 1. 产品定位

**一句话**：模型对当前工作局面输出判断，系统承接并执行。

**核心价值**：不是"系统理解世界"，而是"模型输出可被承接的判断"。

**设计原则**：
- 先定义模型输出，不先定义应用场景
- 系统不拥有语义判断，只提供输入、消费输出、验证结果
- 任何"判断内容"先写成模型输出，不写成系统事实
- 场景只作为验证样本桶，不作为产品对象

---

## 2. 元语言四段式

### 2.1 可观察输入

系统只负责提供模型能观察的输入信号：

| 信号类型 | 说明 |
|----------|------|
| 窗口截图 | 屏幕/窗口像素 |
| 窗口元数据 | 标题、进程名、类型标识 |
| 界面结构 | DOM/UI元素（如果可获取） |
| 历史判断记录 | 过去同类局面的模型输出（离线材料） |

**系统不负责**：理解场景、推断用户意图、判断任务阶段。

---

### 2.2 模型输出内容（第一性对象）

模型必须表达的判断内容，共五项：

| 输出项 | 类型 | 说明 |
|--------|------|------|
| `judgment` | string | 模型对当前局面的判断 |
| `next_step` | string | 模型认为的下一步 |
| `level` | enum | 建议级别：suggest / request / execute |
| `confidence` | float | 判断把握 0.0-1.0 |
| `grounds` | string | 判断依据（不是推理过程，是依据摘要） |

**level 定义**：

| 级别 | 含义 |
|------|------|
| `suggest` | 模型认为有下一步，但只通知用户 |
| `request` | 模型建议执行，需用户确认 |
| `execute` | 模型高置信度，建议直接执行 |

**系统承接元信息**（不是模型输出，放在系统侧）：

```
timestamp / window_id / model_output / system_action / user_feedback
```

---

### 2.3 系统承接

系统消费模型输出，执行对应动作：

```
模型输出 judgment + next_step + level + confidence + grounds
        ↓
系统判断：level 是 suggest / request / execute
        ↓
suggest   → 推送通知给用户
request  → 等待确认 → 执行或放弃
execute  → 调用执行层 → 结果记录
```

**系统只负责**：
- 把模型输出路由到对应处理层
- 执行结果记录
- 执行失败或用户拒绝时的处理

**系统不负责**：
- 修改模型的判断内容
- 在模型未输出的情况下主动行动

---

### 2.4 降级规则

| 情况 | 系统行为 |
|------|----------|
| 模型无判断输出 | 静默等待下一轮，不通知用户 |
| 用户拒绝 request | 记录拒绝，不重试 |
| 执行失败 | 记录原因 |
| 连续失败 N 次 | 暂停该类窗口 |

---

## 3. 关于"无判断"合法输出

模型可以输出"无判断"，即当前局面不足以形成可承接判断。

这种情况下：
- 模型输出空白 judgment，或明确标注 "no judgment"
- 系统静默等待下一轮
- 不通知用户，不记录为失败

---

## 4. 行为日志

**定位**：离线评估材料，不是系统校准机制。

**记录内容**：

```
timestamp / model_judgment / model_next_step / model_level / model_confidence / model_grounds / system_action / user_feedback
```

**用途**：
1. **评估**：模型输出是否被用户接受
2. **回看**：用户可查询历史判断
3. **改进**：供后续批次评估使用

**边界**：行为日志不是模型的真理源，不参与实时判断流程。

---

## 5. 第一阶段范围（验证目标）

### 范围

- **输入**：窗口截图 + 元数据
- **输出**：模型判断内容（judgment + next_step + level + confidence + grounds）
- **系统**：承接路由 + 日志记录
- **不做**：执行层、历史模式匹配、实时校准、跨会话学习

### 验证问题

1. 模型能否对当前局面输出具体而非泛泛的 judgment
2. 模型输出的 next_step 是否对用户合理
3. 模型建议的 level 是否符合用户预期
4. grounds 是否能帮助用户理解判断依据
5. "无判断"输出是否真实、合理

### 验证样本桶

| 类型 | 用途 |
|------|------|
| 内容编辑型窗口 | 文档编辑器、草稿、发布前 |
| 表单/流程型窗口 | 配置页、提交页、审批页 |
| 停滞/待续型窗口 | 中断的工作页、待续教程 |

---

## 6. 交互设计

极简：
- **启动** → 系统开始运行
- **收到 suggest** → 查看通知，可忽略
- **收到 request** → 确认或拒绝
- **execute** → 无感执行

---

## 7. 成功标准

**输出成功**：模型的 judgment + next_step 对用户来说"是此刻合理的下一步"。

**主动成功**：level 符合用户直觉。

**产品成功**：
> 模型不是在理解世界，而是在输出可承接的判断。

---

## 8. 失败信号

- judgment 是泛泛建议，不是对当前局面的具体判断
- next_step 像聊天总结，不是行动假设
- 系统在模型输出之外自行判断了
- 验证对象又退回到"场景是否成立"
- 文档把"历史校准""意图推断"写成系统主能力

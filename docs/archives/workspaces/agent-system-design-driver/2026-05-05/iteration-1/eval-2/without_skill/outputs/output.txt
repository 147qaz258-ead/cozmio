# 多 Agent 架构设计方案

## 当前系统背景

Cozmio 是一个 Tauri 桌面应用，核心流程是：

1. **WindowMonitor** 轮询前台窗口，获取截图 + 元数据
2. **ModelClient** 将 screenshot + window_info 发送给 Ollama API，获取 judgment
3. **Executor** 根据 judgment 决定执行什么动作（通知、弹窗等）

整个流程是单 agent（一个 Vision Model）驱动的：**输入 = 窗口截图 + 元数据，输出 = judgment**。

---

## 为什么需要多 Agent？

当前单 agent 的局限性：

- **单一职责混杂**：一个模型同时要做"理解窗口内容"、"判断是否干预"、"生成回复策略"，导致 prompt 膨胀、决策质量下降
- **无法处理复杂决策链**：有些场景需要"先判断要不要管，再判断谁来管，再判断怎么执行"
- **缺乏专业性**：通用视觉模型在特定领域（窗口监控、任务切换判断）上不如专用 agent
- **无法并行感知多路信号**：比如同时监控多个窗口、同时处理前台交互和后台任务
- **错误难以分层隔离**：一旦判断错误，难以定位是感知层的问题还是决策层的问题

---

## 推荐的多 Agent 分工方案

### 方案 A：流水线式多 Agent（推荐用于 Cozmio）

```
┌─────────────────────────────────────────────────────┐
│  Supervisor Agent (总控)                            │
│  - 负责任务分发、结果汇总、最终决策                 │
└──────────────┬──────────────────────────────────────┘
               │
       ┌───────┼───────┬────────────┐
       ▼       ▼       ▼            ▼
   ┌──────┐ ┌──────┐ ┌────────┐ ┌──────┐
   │Percep│ │Context│ │Decision │ │Action│
   │Agent │ │Agent  │ │ Agent   │ │Agent │
   └──────┘ └──────┘ └────────┘ └──────┘
     感知     上下文     决策        执行
   - 窗口    - 历史     - 判断是否  - 生成具体
     截图      记忆      干预        操作指令
     解析    - 任务链   - abstain   - 通知/
   - 关键     理解       策略       弹窗内容
     元素提                                    │
       取                                      ▼
                        ┌──────────────────────────┐
                        │  Human-in-the-Loop       │
                        │  (用户确认 / 取消)       │
                        └──────────────────────────┘
```

#### 各 Agent 职责详解

**1. Perception Agent（感知 Agent）**
- 输入：原始窗口截图 + 窗口元数据（title, process_name, rect）
- 输出：结构化的"窗口状态描述"
  - 识别窗口类型（浏览器/IDE/文档/游戏/工具等）
  - 提取关键视觉元素（按钮、表单、代码块、文本区域）
  - 标注显著变化（新窗口弹出、焦点切换、弹窗出现）
  - 生成简短语义描述（"用户在 VSCode 中编辑 Rust 代码"）
- 特点：轻量、只做视觉理解，不做决策

**2. Context Agent（上下文 Agent）**
- 输入：Perception Agent 输出 + 历史记忆
- 输出：带上下文的"情境理解"
  - 用户当前任务是什么（基于历史行为链推断）
  - 最近的操作序列（窗口切换频率、任务切换模式）
  - 当前 session 的主要目标（如果已知）
  - 相关的历史经验（之前在类似场景下是如何处理的）
- 特点：依赖 memory 模块，擅长跨时间窗口推理

**3. Decision Agent（决策 Agent）**
- 输入：Perception 输出 + Context 输出
- 输出：
  - `Intervene` / `Abstain` / `Defer`
  - 干预理由（简短的推理链）
  - 置信度评分
- 特点：这是核心的专业判断 agent，需要精心设计的 prompt 和 few-shot examples

**4. Action Agent（执行 Agent）**
- 输入：Decision Agent 输出 + 原始上下文
- 输出：具体的执行指令
  - 通知内容（生成友好的中文提示）
  - 弹窗文案
  - 后续建议（"建议切换到 X 窗口处理"）
- 特点：轻量、专注于"如何表达"，不负责判断"要不要表达"

**5. Supervisor（总控 Agent）**
- 输入：所有 Agent 的中间结果
- 输出：最终决策 + 执行
- 职责：
  - 决定是否需要 Human-in-the-Loop（高风险操作需用户确认）
  - 处理 Agent 之间的冲突
  - 超时控制（某个 agent 响应慢时降级策略）

#### 数据流示例

```
[窗口截图 + 元数据]
        │
        ▼
┌───────────────────┐
│ Perception Agent  │ → "用户正在 VSCode 中编辑 cozmio_core/src/lib.rs"
└───────────────────┘
        │
        ▼
┌───────────────────┐
│ Context Agent     │ → "用户在连续 30 分钟内高频切换窗口，疑似任务切换混乱"
└───────────────────┘
        │
        ▼
┌───────────────────┐
│ Decision Agent    │ → Intervene (置信度 0.85): "检测到高频任务切换，建议整理"
└───────────────────┘
        │
        ▼
┌───────────────────┐
│ Action Agent      │ → "您最近切换窗口比较频繁，需要我帮您整理一下任务吗？"
└───────────────────┘
        │
        ▼
[Human-in-the-Loop 确认] → [执行/取消]
```

### 方案 B：并行感知 + 层次决策

适用于更复杂的多窗口监控场景：

```
        ┌─────────────────────────────────────┐
        │          Orchestrator               │
        │  (任务编排 + 最终决策)               │
        └────────────────┬──────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
   ┌─────────┐      ┌─────────┐      ┌─────────┐
   │Window A │      │Window B │      │Window C │
   │Monitor  │      │Monitor  │      │Monitor  │
   │Agent    │      │Agent    │      │Agent    │
   └────┬────┘      └────┬────┘      └────┬────┘
        │                 │                 │
        └─────────────────┼─────────────────┘
                          ▼
                 ┌─────────────────┐
                 │  Decision Hub   │
                 │  (综合判断)      │
                 └────────┬────────┘
                          ▼
                 ┌─────────────────┐
                 │  Action Router  │
                 │  (分发到对应执行) │
                 └─────────────────┘
```

## 技术实现建议

### 消息传递

使用内部消息队列（如 `tokio::mpsc` 或 `crossbeam-channel`）在 Agent 之间传递结构化消息：

```rust
// Agent 之间的消息格式
enum AgentMessage {
    Perception { window_id: u64, description: String, elements: Vec<VisualElement> },
    Context { session_context: SessionContext, history_summary: String },
    Decision { intervene: bool, confidence: f32, reason: String },
    Action { notification: String, buttons: Vec<ActionButton> },
}
```

### Agent 编排策略

| 策略 | 适用场景 | 复杂度 |
|------|---------|--------|
| 顺序流水线（Pipeline） | 简单场景，固定顺序 | 低 |
| 并行感知 + 层次决策 | 多窗口监控 | 中 |
| 基于优先级的中断 | 高频轮询 + 低频深度分析 | 高 |
| Supervisor 动态调度 | 通用复杂场景 | 高 |

### 错误处理和容错

- **超时降级**：某个 agent 超过阈值时，直接跳过该阶段使用默认值
- **置信度门限**：Decision Agent 置信度低于阈值时，强制 abstain
- **人工兜底**：高风险操作（系统修改、文件操作）必须 human-in-the-loop

## 对 Cozmio 的具体建议

考虑到 Cozmio 的实时性要求（轮询间隔 3 秒），推荐采用**方案 A 的简化版**：

1. **Perception + Decision 合并为一个 Agent**（减少网络延迟）
2. **Context Agent 独立**（利用现有的 memory 模块）
3. **Action Agent 独立**（生成通知文案）

```
[截图 + 元数据]
       │
       ▼
┌─────────────────┐     ┌──────────────┐
│ Vision+Decision │ ←→ │ Context Mem  │
│ Agent           │     │ (现有 memory) │
└────────┬────────┘     └──────────────┘
         │
         ▼
┌─────────────────┐
│ Action Agent   │
│ (生成通知文案)  │
└─────────────────┘
```

这样的好处：
- 减少与 Ollama 的往返次数（延迟关键路径上只有 1-2 次 API 调用）
- 复用现有的 memory 模块，不需要大幅重构
- 容易逐步引入：先分离 Action Agent，验证后再分离 Context Agent

---

## 潜在风险

1. **延迟增加**：多 Agent 串联会增加端到端延迟，需要严格的超时控制
2. **信息丢失**：Agent 之间传递结构化信息时，如何设计格式避免语义丢失
3. **调试复杂度**：多 Agent 调试比单 Agent 复杂，需要完善的日志和 tracing
4. **资源消耗**：多个 Agent 并行会增加内存和计算资源消耗

---

## 下一步行动建议

1. **短期**：先在现有架构中分离出 Action Agent（改动最小）
2. **中期**：引入 Context Agent，复用现有 memory 模块
3. **长期**：根据实际效果决定是否拆分 Perception 和 Decision Agent

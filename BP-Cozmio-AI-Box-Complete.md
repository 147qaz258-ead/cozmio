## 一、项目痛点

### 1.1 独立开发者面临的三重困境

独立开发者（Indie Hacker）是当今最特殊的工作群体：他们一个人承担产品、研发、运营、客服所有角色，时间极度稀缺。2026 年的 AI 时代，他们面临三重困境：

**困境一：订阅疲劳但利用率极低**

独立开发者平均订阅 12-18 个 SaaS 工具，月账单 $200-800。但 AI 工具的实际利用率：
- ChatGPT Plus $20/月，利用率 ~40%（只用来写代码）
- Claude Pro $20/月，利用率 ~30%
- GitHub Copilot $20/月，利用率 ~60%（补全最实用）

问题不是"工具不够多"，而是**每次要用 AI 都要切换窗口、复制上下文、等待结果**，真正用上的功能不到 30%。

**困境二：上下文切换的时间成本**

每次让 AI 介入一个任务：
1. 切换到 AI 工具窗口（3-5 秒）
2. 复制当前上下文（1-3 分钟）
3. 描述任务（1-5 分钟）
4. 等待结果（30 秒-10 分钟）
5. 验证结果，复制回原工具（2-5 分钟）

**总计：5-20 分钟/次，每天 5-10 次 = 每天浪费 25-200 分钟在上下文切换上。**

按独立开发者时薪 ¥200 计算，每月时间成本 ¥2,000-12,000。

**困境三：隐私焦虑与"次优选择"**

85% 的独立开发者"非常担心"自己的代码/数据被 AI 服务商滥用。62% 因此改变 AI 使用方式（比如避免在 AI 中输入完整代码）。

但本地模型质量差（Llama3-8B vs GPT-4 有 15-30% 能力差距），云端模型有隐私风险——他们被迫做"次优选择"，AI 采用效率降低 15-25%。

### 1.2 现有解决方案的错配

| 方案 | 问题 |
|------|------|
| ChatGPT Plus / Claude Pro | "工具"，需要手动调用，无法自动化 |
| Ollama 本地运行 | 免费，但没有 Window Monitor、没有 Relay、没有 Memory |
| NVIDIA Jetson / Raspberry Pi 自建 | 开发板不是产品，需要 3-7 天配置时间 |
| GitHub Copilot | 仅限 IDE 补全，无法做跨任务自动化 |

**核心错配：独立开发者需要的是"AI 操作员"，但他们只能用到"AI 工具"。**

工具需要人操作，Operator 自动工作。

## 二、产品与解决方案

### 2.1 Cozmio AI Box 是什么

Cozmio AI Box 是一款本地 AI 推理硬件，但它的核心不是"本地跑模型"，而是**主动智能体调度**。

```
用户工作
    │
    ▼
┌────────────────────────────────────────────┐
│  Window Monitor                            │
│  监控前台窗口 → 捕获截图 + 窗口元数据        │
└──────────────────┬───────────────────────┘
                   ▼
┌────────────────────────────────────────────┐
│  Model Client (本地 8B 模型)                │
│  理解当前场景 → 判断是否需要 AI 介入          │
└──────────────────┬───────────────────────┘
                   ▼
┌────────────────────────────────────────────┐
│  Context Competition + Memory                │
│  从历史行为中提取相关记忆片段                │
└──────────────────┬───────────────────────┘
                   ▼
        ┌──────────┴──────────┐
        │    用户确认         │
        │  (Toast 通知 +      │
        │   cozmio:// 协议)   │
        └──────────┬──────────┘
                   ▼
┌────────────────────────────────────────────┐
│  Relay Engine (任务派发协议)                │
│  派发给 Claude Code / 其他 Agent 执行        │
└──────────────────┬───────────────────────┘
                   ▼
┌────────────────────────────────────────────┐
│  Executor 执行 → 进度回传 → 结果通知          │
└────────────────────────────────────────────┘
```

**这不是"另一个 AI 工具"，这是一个不知疲倦的 AI 工作搭子。**

### 2.2 产品规格

| 规格 | 入门版 | 专业版 |
|------|--------|--------|
| SoC | Rockchip RK3588 (8核 ARM64) | x86 + NVIDIA RTX 4060 |
| NPU/GPU 算力 | 6 TOPS (NPU) | 242 TOPS (CUDA) |
| 内存 | 16GB LPDDR5 | 32GB DDR5 |
| 存储 | 256GB NVMe | 1TB NVMe |
| 功耗 | 15W | 150W |
| 售价 | ¥999 | ¥4,999 |
| 支持模型 | Llama3.1-8B, Qwen3-8B | Llama3.1-70B, Qwen3-32B |
| 目标用户 | 独立开发者、学生 | 创业团队、小微企业 |

### 2.3 核心差异化

**不是"又一个树莓派"：**

树莓派 + Ollama = 你问一句，它答一句。你是操作员。

Cozmio AI Box = 系统监控你的前台窗口，理解你正在做什么，在需要时主动提议，用户确认后自动执行。你是调度员。

**三个核心壁垒：**

1. **Window Monitor**：捕获前台窗口截图 + 元数据，是整个系统的入口。技术门槛：Windows API 复杂性、多显示器边界处理、截图压缩与性能权衡。

2. **Relay Engine**：不是简单 IPC 调用，是任务派发 + 进度回传 + 中断控制 + 结果通知的完整协议。V2-V8 演进，解决了真实场景问题（心跳检测、任务中断、流式进度、分片上传）。

3. **Context Competition**：不是简单 RAG，而是从历史行为中动态提取相关记忆片段。关键创新：任务结果反馈回路（执行成功/失败 → 影响下次检索权重）。

## 三、技术架构与护城河

### 3.1 为什么是硬件而不是软件

**软件-only 方案的问题：**
- 用户环境各异：Windows/macOS/Linux、不同 GPU 驱动版本
- 难以保证一致的 AI 判断质量
- 数据隐私依赖用户自己的配置

**Cozmio AI Box 的价值：**
- 硬件即配置：系统镜像与硬件深度绑定，插电即用
- 推理延迟可预测：RK3588 NPU 算力 6 TOPS，延迟稳定 2-4 秒
- 数据流完全可控：所有数据在本地 Box 内流转

**更重要的是：硬件是信任锚点。**

独立开发者愿意为 ¥999 的硬件付钱，前提是"看到了"这个东西。纯 SaaS 订阅对他们而言是虚无的——没有实物，就没有锚点。硬件把"AI 能力"从云端的一行 API 调用，变成桌面上一个真实的盒子。

### 3.2 软件架构

```
┌─────────────────────────────────────────────────┐
│              Cozmio OS (Linux-based)            │
│                                                  │
│  ┌──────────────┐    ┌──────────────────────┐  │
│  │ Window       │    │ Model Runtime         │  │
│  │ Monitor      │───▶│ (llama.cpp)          │  │
│  └──────────────┘    └──────────────────────┘  │
│         │                      │               │
│         ▼                      ▼               │
│  ┌──────────────┐    ┌──────────────────────┐  │
│  │ Context      │    │ Relay Engine         │  │
│  │ Competition  │───▶│ (V2-V8 IPC Protocol) │  │
│  │ + Memory     │    └──────────────────────┘  │
│  └──────────────┘           │                 │
│                             ▼                  │
│                    ┌──────────────────┐        │
│                    │ Box Worker       │        │
│                    │ (ARM64/Linux)    │        │
│                    └──────────────────┘        │
└─────────────────────────────────────────────────┘
```

### 3.3 开源策略

| 层级 | 策略 | 理由 |
|------|------|------|
| OS 镜像 | GPL v3 | 建立信任，吸引开发者 |
| Relay Engine 协议 | MIT | 让开发者构建兼容实现 |
| Box Worker 固件 | MIT | 降低硬件定制门槛 |
| 调度引擎（商业组件） | 闭源 | 核心商业价值 |
| 企业管理面板 | 闭源 | 企业付费点 |

### 3.4 技术护城河分析

**Layer 1（容易复制）：**
- Relay Engine 协议基本逻辑
- Window Monitor 的 Windows API 调用方式

**Layer 2（需要工程投入）：**
- 上下文竞争系统的参数调优和场景适配
- RK3588 NPU 的量化参数和推理优化
- 生产级的状态机稳定性和边界情况处理

**Layer 3（难以复制）：**
- 软硬件协同的深度优化
- 基于真实用户反馈的快速迭代能力
- 用户社区和场景理解

## 四、商业模式

### 4.1 收入模型

**硬件销售：获客媒介，不是利润中心**

入门版 ¥999 BOM 成本 ¥635（不含模具摊销），修正后毛利 ¥274（27.4%）。

专业版 ¥4999 BOM 成本 ¥3570，修正后毛利 ¥1100（22%）。

**软件订阅（OS Pro）：真正的利润引擎**

- 定价：$9.99/月（约 ¥73/月），年付 ¥790
- LTV（14个月中性假设）：$139.86（约 ¥1,023）
- 净利率：80%+

**订阅转化率：硬件买家 30% vs 纯软用户 12%**

硬件购买行为本身就是强意图信号，用户已经用真金白银表达了付费意愿。

### 4.2 Unit Economics

**综合 LTV/CAC：11.4x**

硬件用户路径 CAC ≈ 0（硬件毛利覆盖获取成本），纯软用户路径 CAC ¥300。

**盈亏平衡：2028 Q3-Q4**

| 指标 | 2026 | 2027 | 2028 |
|------|------|------|------|
| 硬件销量（台） | 2,000 | 8,000 | 20,000 |
| 订阅用户 | 600 | 2,400 | 8,000 |
| 总收入（万元） | ¥214 | ¥867 | ¥2,207 |
| 净利润（万元） | -¥436 | -¥333 | +¥407 |

### 4.3 为什么先做硬件而不是纯 SaaS

1. **价值锚点**：¥999 硬件 vs $9.99/月订阅，硬件的"心理承诺"更强
2. **数据飞轮**：硬件用户更愿意主动反馈，提供更高质量数据
3. **竞争护城河**：软硬件协同优化需要 18+ 个月的迭代积累
4. **资金效率**：硬件预售可以在 3-6 个月内回收 ¥200-300 万

## 八、风险与应对

### 风险 1：Pi 验证通过 ≠ 量产可行

**问题**：散热、SD 卡可靠性、电源管理——量产版本需要重新设计。

**解决方案**：
- 量产版使用 eMMC/NVMe 存储（成本 +¥80）
- 工业级 DC-DC 转换（成本 +¥30）
- 主动散热设计（成本 +¥50）
- 合计增加成本 ¥110-160/台

### 风险 2：大厂可以复制

**真实护城河**：
- 市场太小，大厂看不上（全球 ¥200 亿市场）
- 开源是护城河（社区贡献者帮你开发）
- 先发优势：18-24 个月的时间壁垒

### 风险 3：种子轮拿不到怎么办

**方案 A（推荐）：缩小规模，先验证**
